# Oracle Council: L-4 CLI Adapter Spike Report

- 対象仕様: `SPEC.md` §13.6 (Phase 0前spike: Adapter capability確認)
- 実施日: 2026-07-12
- 検証環境: Windows 11 / PowerShell 7.4.x / CMD

---

## 1. Claude Code (2.1.206) 検証結果

### 1.1. 完全非対話で1回の呼び出しを実行できるか
* **検証コマンド**: `cmd /c "claude -p ""Say hello"" < NUL"`
* **判定**: **可能**。`-p` または `--print` オプションを指定し、`stdin` を `NUL`（または `/dev/null`）からリダイレクトすることで、一切のプロンプト対話なしで1回の応答のみを出力して正常終了することを確認。

### 1.2. ツール実行の無効化または拒否を保証できるか
* **検証オプション**: `--tools ""`
* **判定**: **保証可能**。`--tools ""` オプションを与えることで、ビルトインの全ツール（Bash、Edit、Readなど）が無効化され、AIエージェントはツール（コマンド実行やファイル編集）を使わずにテキスト応答のみを生成する。

### 1.3. 読み取り専用または空の一時cwdで実行できるか
* **判定**: **可能**。ファイルシステム書き込み権限のないディレクトリや、完全に空のテンポラリディレクトリ下でも、ローカル設定ファイルがないことを検知しつつも正常に動作する。また `--no-session-persistence` を指定することで、実行履歴をディスクに保存せずに使い捨て実行が可能。

### 1.4. 構造化出力（JSON）を安定して取得できるか
* **検証オプション**: `--output-format json`
* **判定**: **可能**。`--output-format json` を指定して非対話実行（`-p`）を行うと、結果が単一のJSON形式で stdout に出力される。

### 1.5. stdout / stderr / exit code の分離と安定性
* **判定**: **安定**。
  * 正常応答は stdout へ出力される。
  * 進行状況や警告は stderr へ出力される。
  * セッション制限（Quota制限など）や実行時エラーが発生した場合は、`You've hit your session limit` 等のエラーメッセージを出力し、非ゼロ（exit 1）で終了する。この文字列パターンを検知することで `QUOTA_EXCEEDED` などのステータスへ正規化可能。

### 1.6. seed指定の可否
* **判定**: **不可**。Claude Code の CLI オプションには `seed` または `temperature` などのパラメータを直接指定するオプションは存在しない。
* **対応**: `SPEC.md` §7.5 のシード指定に関する仕様は「対応するCLI（実動時にオプションを持つもの）のみに限定して適用する」とする。

## 2. Codex CLI (0.144.0) 検証結果

### 2.1. 完全非対話で1回の呼び出しを実行できるか
* **検証コマンド**: `cmd /c "codex exec ""Say hello"" --ephemeral < NUL"`
* **判定**: **可能**。`codex exec` サブコマンドを使用し、`stdin` を `NUL` からリダイレクトすることで非対話実行可能。`--ephemeral` を指定することでセッションファイルをディスクに永続化させずに使い捨てで実行可能。

### 2.2. ツール実行の無効化または拒否を保証できるか
* **検証オプション**: `-s read-only` または `--sandbox read-only`
* **判定**: **保証可能**。サンドボックスポリシーとして `-s read-only` を指定することで、モデル生成のシェルコマンド実行時にファイル編集や書き込みを遮断・拒否し、読み取りのみに制限可能。

### 2.3. 読み取り専用または空の一時cwdで実行できるか
* **判定**: **可能**。Claude Code と同様、空のディレクトリでも正常に起動し動作することを確認。

### 2.4. 構造化出力（JSON）を安定して取得できるか
* **検証オプション**: `--output-schema <FILE>`
* **判定**: **可能**。期待する出力スキーマ（JSON Schemaファイル）を指定することで、最終応答のフォーマットを強制的にJSON化し、かつその構造を保証させることが可能。

### 2.5. stdout / stderr / exit code の分離と安定性
* **判定**: **安定**。
  * `exec` による最終出力は stdout の末尾に JSON 形式で出力され、パースが容易。
  * 実行メタデータや承認時のプロンプト警告などは stderr または stdout のヘッダ部に分離されている。
  * 異常終了時は非ゼロの終了コードが返される。

### 2.6. seed指定の可否
* **判定**: **不可**。`seed` を指定する直接のオプションは提供されていない。
* **対応**: Claude Code と同様、対応対象外（または将来対応）とする。

## 3. 総括と実装方針へのフィードバック

1. **安全性（Tool無効・Read-Only）の担保**:
   * 両 CLI ともに、ツールの無効化（Claude: `--tools ""`, Codex: `-s read-only`）および非対話化（`-p` / `exec`）が提供されており、安全な実行（ファイル改ざんや任意コード実行の防止）が `fail-closed` で担保可能である。
2. **プローブと能力**:
   * `probe()` 時には、実際に `--version` や極めて短いプロンプト（"test"）を投げてプローブし、エラーメッセージ（`session limit` や `AUTH_REQUIRED` 関連メッセージ）を検知してステータス（`QUOTA_EXCEEDED` / `AUTH_REQUIRED`）にマッピングする。
3. **データフォーマットのパース**:
   * 両CLIとも JSON 出力をサポートしているため、パースエラー時は `INVALID_OUTPUT`（exit 1）として安全に処理を中断する。
