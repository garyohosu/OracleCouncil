# AIの答えを、そのまま信じなくていい仕組みを作りたい。「Oracle Council」開発記

> ステータス: 開発中の記事草稿  
> 最終更新: 2026-07-10  
> 開発の進行に合わせて内容を更新する。

## AIは便利になった。でも、最後に困ることがある

AIに質問すると、驚くほど自然な文章で答えてくれる。

料理、旅行、パソコンの不具合、仕事の調査、健康の心配。以前なら検索結果を何ページも読んでいたことでも、AIなら数秒で整理してくれる。

ところが、最後にひとつ困ることがある。

> **この答え、本当に合っているのだろうか。**

AIは、分からないことでもそれらしい文章を作ることがある。日付を間違えたり、存在しない資料を紹介したり、古い情報を現在の情報として説明したりする。

文章が自然なので、間違いに気づきにくい。ここが少々厄介だ。

「AIの回答は必ず確認しましょう」と言われても、確認するために何時間も検索するなら、最初から自分で調べるのとあまり変わらない。

そこで、こんなソフトを作れないかと考えた。

> **質問を入れるだけで、複数のAIが回答を確認し、外部の資料まで調べてから返してくれるソフト。**

名前は「Oracle Council（オラクル・カウンシル）」にした。日本語にすれば「神託の評議会」くらいの意味になる。

ただし、神託をありがたく受け取るだけの評議会ではない。互いの答えを疑い、証拠を探し、分からないことは分からないと言わせる。かなり疑り深い評議会である。

## 最初は「4つのAIに相談させればよい」と考えた

最初の案は単純だった。

```text
質問する
  ↓
4つのAIが別々に答える
  ↓
AI同士で意見を出し合う
  ↓
みんなが納得した回答を返す
```

1つのAIだけに聞くより、4つに聞いたほうが間違いを見つけやすそうだ。人間でも、ひとりで考えるより複数人で確認したほうが安心できる。

ところが、設計を始めてすぐに問題が見つかった。

**4つのAIが同じ答えを出しても、その答えが正しいとは限らない。**

4つとも同じ古い情報を覚えていたら、全員一致で間違える。同じ有名な誤解を参照していたら、堂々と全会一致になる。

多数決で決められるのは「人気のある答え」であって、「正しい答え」ではない。

そこで、Oracle Councilの中心を変えた。

> AIの数ではなく、答えを裏付ける根拠を優先する。

AI同士の議論は、そのための手段のひとつにすぎない。

## 一般の人が欲しいのは、AI会議の議事録ではない

技術者としては、AI同士が反論する様子を見るだけでも面白い。

しかし、普通に使う人が欲しいのは、AIたちの長い会議録ではないだろう。

知りたいのは、たぶん次の3つだ。

- 結局どうなのか
- どこまで確認できたのか
- 確認できなかった部分は何か

Oracle Councilでは、表側をできるだけ簡単にしたい。

```text
質問を入力
  ↓
必要なら質問の意味を確認
  ↓
複数のAIが別々に回答
  ↓
外部資料で重要部分を確認
  ↓
別のAIが最終回答を監査
  ↓
確認状況と一緒に回答を表示
```

利用者は普通の言葉で質問するだけ。内部で誰が回答し、誰が批評し、どの資料を調べたかは、必要な人だけが詳細画面で確認できるようにする。

## 質問する側が、条件を全部そろえられるとは限らない

AIの間違いばかり注目されるが、質問そのものが不完全な場合もある。

たとえば、

> おすすめのパソコンは？

と聞かれても、用途や予算が分からなければ答えは大きく変わる。

一方、

> 富士山の標高は？

なら、そのまま調べて答えられる。

何でも聞き返すAIは面倒だ。しかし、必要な条件を確認せずに勝手な前提で答えるAIも困る。

そこでOracle Councilでは、質問を3種類に分ける。

- そのまま答えられる
- 前提を明示すれば答えられる
- 追加情報がないと結論が変わる

私は以前、「対話型プロンプトメーカー」というGPTを作った。曖昧な希望を少しずつ聞きながら、よりよい依頼文へ整えていくものだ。これまでに約5万人が利用している。

その経験から分かったのは、一般の利用者が欲しいのは「完璧な質問の書き方講座」ではなく、**普通の言葉から意図をくみ取ってくれる仕組み**だということだ。

Oracle Councilにも、この考え方を取り入れる。ただし、質問攻めにはしない。結論を左右することだけを聞く。

## 利用上限になったAIがいても、残りで続ける

複数のAIを使うと、いつも全員が参加できるとは限らない。

利用上限、認証切れ、混雑、応答時間切れなどが起きる。ひとつのAIが使えないだけで全体が止まってしまうと、実用にならない。

そこで、参加できないAIはその回だけ欠席扱いにする。

```text
Claude Code  参加
Codex CLI    参加
別のAI       利用上限のため欠席
```

残ったAIで処理を続ける。ただし、参加者が少なかったことは利用者に明示する。

全員がそろうまで会議を始めない仕組みでは、AIより先に人間が帰ってしまう。

## 「分かりません」を失敗にしない

このソフトで最も大切にしたいことがある。

それは、確認できないときに無理に答えないことだ。

Oracle Councilの結果は、単純な「正解」「不正解」ではなく、次のように分ける。

- 確認できた
- 一部だけ確認できた
- 確認できなかった
- 信頼できる資料同士が食い違った
- 回答を保留した

AIは何か答えると賢そうに見える。しかし、分からないのに答えを作るより、「ここから先は確認できません」と止まれるほうが役に立つ場面は多い。

特に、健康、お金、法律、安全に関わる質問では、この違いが大きい。

## 設計を始めたら、AI会議が大きくなりすぎた

ここまでは、聞こえのよい話である。

実際に設計してみると、すぐに別の問題が出てきた。

最初の仕様では、4つのAIが回答し、全員が批評し、最後に全員が投票する予定だった。慎重にしようと機能を足していった結果、ひとつの質問に対して最大23回もAIを呼び出す設計になった。

これでは、正確な回答を得る前に時間と利用枠を使い切ってしまう。

そこで、設計を根本から見直した。

- 本当に全員で批評する必要があるのか
- 投票は品質を上げるのか
- 何を外部資料で確認すべきか
- どこで処理を打ち切るべきか
- 質問や回答を保存してよいのか
- AIにパソコン内のファイルを触らせずに動かせるか

複数のAIに仕様書を読ませ、反論させ、また直した。その結果、通常のAI呼び出しを7回まで減らす案へ変わった。

単に機能を追加するより、**何を捨てるかを決めるほうが難しい**。そして、この削減の過程こそ、今回の開発で最も面白い部分になってきた。

- Claude CodeとCodex CLIを安全に呼び出せるか実際に検証した結果
- ユースケース図、シーケンス図、クラス図を使った設計過程
- 多数決、単独AI、Oracle Councilのどれが本当に正確かを比較する実験

完成したソフトの使い方だけではなく、複数のAIと設計レビューを繰り返しながら、何を採用し、何を捨てたかまで記録する。

「AIをたくさん使えば賢くなる」という単純な話ではない。AIを増やすほど、費用、時間、同調、情報漏えいといった別の問題も増える。
