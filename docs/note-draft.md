# 4つのAIに議論させるだけでは足りない。AIの間違いを自動検証する「Oracle Council」を作り始めた

> ステータス: 開発中の記事草稿  
> 最終更新: 2026-07-10  
> この文書は開発の進行に合わせて更新する。

## 4つのAIに同じ質問をしたら、正しい答えになるのか

AIオーケストレーションの第二弾として、新しいソフトを作り始めた。

名前は「Oracle Council」。最大4つのAI CLIへ同じ質問を送り、それぞれに独立して回答させ、回答を相互に批評し、最終的に合意した回答を返すソフトである。

最初は、単純に次のような仕組みを考えていた。

```text
質問
  ↓
4つのAIが独立回答
  ↓
AI同士で議論
  ↓
合意した回答を返す
```

複数のAIを使えば、1つのAIだけに聞くよりも間違いを見つけやすい。回答の違いそのものも興味深い。

しかし、すぐに大きな問題へ気づいた。

**4つのAIが同じ答えを出しても、その答えが正しいとは限らない。**

4つとも同じ古い情報を学習していたり、同じ有名な誤情報を参照していたりすれば、全会一致で間違える可能性がある。多数決は合意を作れても、真実を保証しない。

そこでOracle Councilの目的を、「AI同士を議論させること」から少し変えることにした。

> 複数のAIが回答を作り、互いの間違いを疑い、外部の根拠を確認してから最終回答を返す。

## 一般ユーザーが欲しいのはAI会議ではない

AI同士の討論は、技術者から見ると面白い。しかし一般ユーザーが本当に欲しいのは、会議の議事録ではないだろう。

欲しいのは、おそらく次のようなものだ。

> 質問を入力するだけで、間違いの少ない回答が返ってくる。

そこで表向きの機能は、できるだけ単純にする。

```text
質問を入力
  ↓
質問の不足や誤った前提を確認
  ↓
複数AIが独立回答
  ↓
事実主張を抽出
  ↓
外部資料で検証
  ↓
AIが根拠を見ながら相互批評
  ↓
統合・監査・投票
  ↓
検証状況付きの回答を表示
```

内部ではかなり複雑な処理を行うが、ユーザーには最終回答と「どこまで確認できたか」を中心に見せる。

AI同士の回答、批評、投票結果、根拠資料は、必要な人だけが「検証の詳細」から確認できる形を考えている。

## そもそも質問が不完全かもしれない

AIの回答が間違う原因は、AI側だけではない。

ユーザーの質問が曖昧だったり、必要な条件が抜けていたり、誤った前提を含んでいたりする場合もある。

例えば、次の質問だけでは答えを決めにくい。

```text
おすすめのパソコンは？
```

用途、予算、持ち運びの有無によって、候補は大きく変わる。

一方で、次の質問なら追加確認はほとんど必要ない。

```text
富士山の標高は？
```

Oracle Councilでは、すべての質問へ機械的に聞き返すのではなく、次の3種類に分ける予定だ。

- そのまま回答できる
- 仮定を明示すれば回答できる
- 追加情報がないと結論が大きく変わる

私は以前、「対話型プロンプトメーカー」というGPTを作った。ユーザーの曖昧な要望を聞きながら、改訂されたプロンプト、改善提案、追加質問を繰り返し、完成度を上げていくものだ。現在までに約5万人に使われている。

Oracle Councilでは、この対話設計をそのまま流用するのではなく、「回答に本当に必要な情報だけを聞く質問整理エンジン」として再設計する。

AIとの会話が便利になるはずなのに、毎回十数問の聞き取り調査が始まったら少し困る。必要以上に賢そうな顔をして質問し続けるAIにはしたくない。

## トークン切れのAIは欠席扱いにする

4つのAI CLIを使う場合、いつも全員が参加できるとは限らない。

利用上限、レート制限、認証切れ、タイムアウト、コンテキスト長の超過などが発生する。

Oracle Councilでは、1つのAIが失敗しただけで全体を停止させない。

```text
Claude   OK
Codex    OK
Gemini   PASS_QUOTA
Agent 4  OK
```

この場合、Geminiはその実行では欠席または棄権扱いとし、残り3つのAIで処理を続ける。

参加数ごとの基本動作は次のように考えている。

- 4 AI: 通常の評議会
- 3 AI: 3つで継続
- 2 AI: 統合役と監査役を分けて継続
- 1 AI: 単独回答・自己監査と明示
- 0 AI: 回答不能

「全員揃うまで会議を始めません」では、AIより先に人間が帰ってしまう。

## EvidenceはOracle Council自身が集める

設計上、特に重要だったのが外部資料の集め方である。

各AI CLIが持つWeb検索機能だけに任せると、AIごとに検索結果や引用方法が異なり、Oracle Council側で本当に根拠が存在するか確認できない場合がある。

そこで、外部資料の収集はOracle Council側の`EvidenceProvider`を正本にすることにした。

```python
class EvidenceProvider(Protocol):
    async def search(self, query: str, *, limit: int) -> list[SearchResult]:
        ...

    async def fetch(self, result: SearchResult) -> EvidenceDocument:
        ...
```

AI CLIの組み込み検索は補助情報として利用できる。ただし、URLや資料識別子を取得できず、Oracle Council自身が内容を確認できないものは「検証済み」の根拠には数えない。

URLが存在するだけでも不十分である。

- そのページへ実際にアクセスできるか
- 主張を裏付ける記述が本当にあるか
- 情報は古くないか
- 引用文は原文と一致するか
- 同じ記事の転載を複数資料として数えていないか

ここまで確認して、初めてEvidenceとして扱う。

## AIの役割は固定しない

Claudeを議長、Geminiを調査役、Codexを批評役のように固定すると、特定モデルへの依存が強くなる。

Oracle Councilでは、役割をAIの職種ではなく、処理フェーズとして扱う。

MVPでは次の分担を予定している。

- Clarifier: 1 AI
- Responder: 利用可能な全AI
- Claim Extractor: 1 AI
- Verifier: 1 AI
- Critic: 利用可能な全AI
- Synthesizer: 1 AI
- Auditor: Synthesizerとは別の1 AI
- Voter: 利用可能な全AI

回答と批評は全員で行う。一方、Claim抽出、検証、統合は1つのAIへ担当させ、最後に別のAIが監査する。

同じAIが複数フェーズを担当することはあるが、前の会話履歴は引き継がない。各フェーズを新しいセッションとして実行し、必要な情報だけを構造化して渡す。

これは、最初に自分が書いた回答へ最後まで執着する「AI版メンツ問題」を減らすためでもある。

## 合意度と検証度は別に表示する

Oracle Councilでは、AIの合意と事実確認を別の指標として扱う。

```text
AI合意: 3 / 3
Claim検証: 5 / 6
結果: 一部検証済み
```

全AIが同意していても、外部資料で確認できなければ「検証済み」にはしない。

逆に、1つのAIが反対していても、信頼できる一次資料が明確に結論を支持していれば、根拠を優先する。

Oracle Councilが目指すのは民主主義ではなく、根拠主義である。

## 分からないときは、無理に答えない

ハルシネーションを完全にゼロにする、と宣言することはできない。

情報源そのものが間違っている場合もある。資料が公開されていないこともある。最新情報を取得できないこともある。複数の信頼できる資料が食い違うこともある。

だからこそ、Oracle Councilには次の回答を許す。

- 検証済み
- 一部検証済み
- 未確認
- 情報が競合
- 回答保留

「分かりません」はAIの敗北ではない。分からないのに、それらしい答えを作ることのほうが問題である。

## 現在の進捗

2026年7月10日時点では、まだ実装前の設計段階である。

GitHubにリポジトリを作成し、`SPEC.md`と`QandA.md`を用意した。仕様レビューでは9カテゴリ・23問の不明点が出たため、役割分担、Evidence収集、タイムアウト、合意ロジック、データ形式、セキュリティ、CLI、テスト方針などを整理した。

主な決定事項は次のとおり。

- Python 3.11以上
- 最大4つのAI CLI
- 既定モードは`verify`
- 保存形式はまずJSONL
- Evidence収集は交換可能なProvider方式
- ライセンスはMIT
- Web UIは将来同じリポジトリへ追加予定

次は、CLIアダプターの共通インターフェースと最小のオーケストレーターから実装を始める。

## これから作るもの

最初のMVPでは、すべてを一気に作らない。

1. 複数AI CLIの起動と状態判定
2. 質問整理
3. 独立回答と相互批評
4. Claim抽出とEvidence検証
5. 統合、監査、投票
6. JSONL保存と実行履歴

この順番で進める予定だ。

最終的には、一般ユーザーが難しいプロンプトを書かなくても、普通の言葉で質問するだけで使えるものにしたい。

> 質問するだけ。AIの間違いチェックは評議会におまかせ。

まだ評議会の建物を設計した段階で、議員は一人も着席していない。これから本当に4つのAIが協力できるのか、それとも全員が好き勝手な演説を始めるのか、実装しながら確かめていく。
